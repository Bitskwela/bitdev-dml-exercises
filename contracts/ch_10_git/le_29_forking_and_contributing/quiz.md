# Quiz: Forking and Contributing

Answer the following questions to check your understanding of this lesson.

## Question 1

What is a fork?

A) A tool to delete repositories
B) A personal copy of someone else's repository on your GitHub account
C) A type of Git branch
D) A merge conflict

**Correct Answer: B**

**Explanation:** Forking creates a copy of a repository under your account, allowing you to make changes without affecting the original project.

---

## Question 2

What is the typical workflow for contributing to an open source project?

A) Edit files directly on the main repository
B) Fork → Clone → Create branch → Make changes → Push → Open PR
C) Just email the maintainers

**Correct Answer: B**

**Explanation:** Fork the repo, clone your fork, create a feature branch, make changes, push to your fork, then open a pull request to the original repository.

---

## Question 3

What is an "upstream" remote?

A) A remote pointing to your fork
B) A remote pointing to the original repository you forked from
C) The default Git server

**Correct Answer: B**

**Explanation:** Convention uses "origin" for your fork and "upstream" for the original repository, letting you sync changes from the source project.

---

## Question 4

True or False: After forking, your fork automatically stays in sync with the original repository.

**Correct Answer: False**

**Explanation:** Forks don't auto-sync. You must manually fetch from upstream and merge or rebase to keep your fork up-to-date.

---

## Question 5

What should you read before contributing to an open source project?

A) Only the code
B) CONTRIBUTING.md and any project guidelines
C) Nothing, just submit code

**Correct Answer: B**

**Explanation:** Projects often have CONTRIBUTING.md files explaining coding standards, PR requirements, and how to submit issues—read these first!
